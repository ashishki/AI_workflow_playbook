# Retrieval and Context Packet Architecture

Version: 1.0
Last updated: 2026-05-25

---

## Retrieval Goal

Retrieval exists to assemble the smallest defensible context for a role and scope. It is not a mandate to search everything.

The system optimizes for:

- deterministic recall before semantic recall
- explicit evidence paths
- bounded context size
- role-specific packets
- reproducible retrieval decisions

---

## Retrieval Modes

| Mode | Source | Use first? | Notes |
|------|--------|------------|-------|
| Task scoped | `Context-Refs` in `docs/tasks.md` | Yes | Highest signal for implementation work |
| State scoped | `docs/CODEX_PROMPT.md` | Yes | Current baseline, next task, open findings |
| Decision scoped | `DECISION_LOG.md`, ADRs | Yes | Architecture and governance changes |
| Evidence scoped | `EVIDENCE_INDEX.md`, evals, tests, reviews | Yes | Regression, proof, and reviewer context |
| Graph scoped | explicit markdown links and manifest edges | Sometimes | Useful for cross-project reuse |
| Semantic scoped | optional vector/BM25 over selected artifacts | Last | Discovery only; must cite canonical files |

## Retrieval Manifest

A retrieval manifest is a deterministic index of repo artifacts. It can be generated by `tools/cognition_index.py` or curated manually when a repo is small.

Minimum fields per artifact:

- `path`
- `artifact_kind`
- `title`
- `canonical`
- `generated`
- `headings`
- `links`
- `sha256`

Generated location:

```text
generated/cognition/index.json
```

Manual project summary:

```text
docs/COGNITION_MANIFEST.md
```

The JSON manifest is optimized for agents and scripts. The markdown manifest is optimized for humans.

## Selection Algorithm

Context packet builders should apply this order:

1. Include role essentials.
2. Include exact task `Context-Refs`.
3. Include open findings affecting the scope.
4. Include decision and ADR lineage affecting the scope.
5. Include eval/evidence artifacts affecting the scope.
6. Include recent journal entries affecting the scope.
7. Include cross-project pattern notes only when explicitly requested or linked.
8. Use semantic retrieval only to propose additional candidates.

Stop when the packet has enough evidence to act. Do not keep adding context because it is available.

## Role Packet Profiles

| Role | Required core | Usually excluded |
|------|---------------|------------------|
| Strategist | architecture, decision log, relevant ADRs, evidence gaps, project fit notes | implementation details unrelated to decomposition |
| Orchestrator | CODEX prompt, tasks, decision log, journal, evidence index, open findings | old archive reviews unless finding-linked |
| Implementer | current task, contract rules, direct dependencies, narrow file scope, relevant eval gate | broad strategy docs and unrelated audit archives |
| Reviewer | contract, task ACs, changed files, evidence index, prior findings, eval artifacts | product brainstorming and unverified research |
| Researcher | research brief, source register, decision consumer, caveats | current implementation state unless decision-linked |
| Postmortem writer | incident timeline, failing checks, related ADRs, prior findings, action evidence | unrelated project history |

## Packet Shape

Context packets are markdown and may include frontmatter:

```yaml
---
artifact_kind: context_packet
role: reviewer
project: lead-response-sla-agent
scope: T21 retrieval query
source_manifest: generated/cognition/index.json
generated: true
canonical: false
created: 2026-05-25
---
```

Required sections:

- `Scope`
- `Role`
- `Canonical Artifacts`
- `Included Context`
- `Excluded Context`
- `Open Findings`
- `Evidence`
- `Decision Lineage`
- `Eval Gates`
- `Next Action`

Packets should cite file paths and short excerpts. They should not paste entire long documents unless the task is explicitly architecture-shaping or high-risk.

## Evidence Lineage Retrieval

When investigating a regression or repeated finding, retrieve in this order:

1. failing test or eval command
2. current eval artifact row
3. previous baseline row
4. evidence index row
5. task that last changed the boundary
6. ADR or decision log row for the boundary
7. prior review report that opened or closed the finding
8. implementation journal entry for the change

The answer must distinguish:

- what failed now
- what previously passed
- which decision authorized the behavior
- which artifact proves the old behavior
- what changed since then

## Semantic Retrieval Policy

Semantic retrieval is optional and supplemental. If enabled:

- index only selected markdown and source files
- store vector indexes under `generated/` or a local cache excluded from Git
- return path citations with every result
- never treat an embedding hit as a decision, finding, or proof
- never include uncited semantic memory in a context packet
- keep deterministic retrieval quality tests for the manifest and packet builder

Semantic retrieval is useful for discovery across large archives, but deterministic links decide what agents may rely on.

## Failure Modes

| Failure | Stop condition |
|---------|----------------|
| Packet contains no canonical evidence for a risky change | Stop and request/add `Context-Refs` |
| Eval regression has no previous baseline | Treat eval memory as incomplete |
| Semantic result has no source path | Exclude it |
| Generated note contradicts ADR | Fix source or regenerate; ADR wins |
| Packet exceeds role budget | Remove lower-priority archives and keep citations |
